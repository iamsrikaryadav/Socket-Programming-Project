Socket programming project: Web Server

Introduction:

This week 2 socket programming project gives you experience in developing concurrent network applications. The client program (the browsers) communicates with the server program (your web server) using HyperText Transport Protocol.

Why a web server?

  1. Web browsing that we experience today is a combination of multiple layers and technologies (such as wifi, ISPs, routing, DNS, reliable transport etc.). At a higher level a webserver is a typical socket program that is capable of handling multiple client requests.
  2. This project helps you think about numerous tasks that a networking application does for making a web server.
  3. This will remove the mystery/magic of what the network does ‘underneath’ your web browsing.
  4. In building this project, you will become familiar with HTTP.

Implementation Requirements Overview:

Web server socket program(s) must recognize HTTP keywords as commands. Your server will implement HEAD, GET (both needed for HTTP 1.1 general purpose server compliance) along with the following requirements:

  • GET – requests a specified resource; it should not have any significance other than retrieval.
  • HEAD – asks for an identical response as GET, without the actual body—no bytes from the requested resource.
  • For all other commands, your server must return “501 Method Unimplemented.”
  • If you are unable to implement one of the above commands (perhaps you ran out of time), your server must return the error response “501 Method Unimplemented,” rather than failing silently (or not so silently).
  • While you develop, you may want to just return this error response always until features are implemented-no matter what you will have a valid HTTP 1.1 server. Your server should be able to support multiple clients concurrently.
  • While the server is waiting for a client to send the next command, it should be able to handle inputs from other clients. Also, your server should not hang if a client sends only a partial request.
  • In general, concurrency can be achieved using either select() or multiple threads. Your code needs to use select() or multiple threads as adding support for multiple clients at once (Note: Develop measures such that you can decide which approach is better for your web server).
  • As a web server, your implementation should be robust to client errors. You are implementing a real, standards-compliant web server.

Step by Step Guide for Checkpoints:

  1. Checkpoint 1: Valid Request Handling:
      a. Echoes to valid requests from Chrome, Firefox and Edge.
        i. Create socket with an IP address and bind it with a port number
       ii. Start server and listen to a client's request.
      iii. On connection requests from clients, accept the connection and start serving the client requests.
      iv. Provide a response as echo of the request made by the client.
  2. Checkpoint 2: Respond properly to any HTTP 1.1 request with HEAD, GET:
     a. Checkpoint 1 Should be completed
     b. Responds to single client
        i. By default supports text as content media type
       ii. On connect default response from server should be index.html page
      iii. Fetch data from the server based on requests made by the client as GET or HEAD.
     c. Provide correct error code to requests made
        i. 400 Bad Request: Server does not recognize the URL. Double check to see that you have typed it correctly.
       ii. 403 Forbidden Pages: You are not being allowed access, for whatever reason. (Example: log page)
      iii. 415 Unsupported Media Type: The server will not accept the request, because the media type is not supported
  3. Checkpoint 3: Display files in directory as hyperlink and trigger GET to fetch data when clicked
     a. Checkpoint 2 Should be completed
     b. Responds to single client
       i.If the index.html page is not present in the server directory, the html page should be created based on files or directories present in the directory as anchor tags to trigger GET when performed click action.
      ii.Should support content media type of Image and PDF
  4. Checkpoint 4: Support multiple clients:
     a. Checkpoint 3 Should be completed
     b. Achieve concurrency
      i. Accepts/replies to requests from concurrent clients at the same time using select or multithreading
  5. Checkpoint 5: Measuring web server performance
     a. Checkpoint 4 Should be completed
     b. Write down how many clients your web server can handle at same time.
     c. How much time does your web server take to serve one client.
     d. Can you improve your web server performance (Time, No of Concurrent Clients)
     e. Maintain a number of TCP connections (States) your web server is handling.
     f. Above metrics can either be recorded in log files or display it on a web server console.
